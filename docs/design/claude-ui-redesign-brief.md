# Claude Design UI Redesign Brief

> 用途：将本文件交给 Claude Design 或其他外部 UI/UX 设计助手，帮助其理解当前产品、页面、功能边界与改版目标。
>
> 重要约束：不要沿用或复刻当前页面的按钮布局位置。请重新思考信息架构、操作区组织和按钮布局。本 brief 只描述功能、操作类型、优先级和状态，不规定按钮应该放在哪里。

## 1. 产品定位

这是一个面向 A 股投资研究的 Web 工作台，当前项目名可理解为 `A_stock_AI`。

它不是营销官网、资讯门户、交易下单系统，也不是投资组合管理工具。当前阶段的核心目标是：

- 让用户输入股票代码或名称，触发 AI 股票分析。
- 让用户查看结构化分析报告，包括技术面、基本面、资金面、新闻情报、策略判断等内容。
- 让用户基于某份报告继续追问 AI Agent。
- 让用户管理模型、数据源、通知渠道、系统参数和 Agent 设置。
- 让整体体验更像专业研究工作台：清晰、克制、高效、可反复使用。

当前产品方向是精简旧功能，聚焦“核心分析链路 + 问股 Agent + 可交互 Web 工作台”。

- 股票分析首页。
- 股票搜索和自动补全。
- 分析任务状态。
- 历史分析记录。

- 问股 Agent 多轮对话。

- 系统设置。
- Agent 设置。
- 登录保护。

### 2.2 不要设计为核心入口

这些能力不是当前核心，不要把它们设计成一级导航或显著入口：

- Bot 机器人通知。
- Electron 桌面端。
- 持仓管理。
- 回测。
- 数据质量面板。
- 营销介绍页。
- 社区、资讯流、课程、交易下单等外延功能。

## 3. 当前页面地图

当前 Web 前端只有少量核心路由：

- `/`：股票分析工作台首页。
- `/chat`：问股 Agent。
- `/settings`：系统设置。
- `/login`：登录页。
- 404 页面：找不到页面时展示。

设计时请保持导航结构简洁。可以优化导航命名、页面分区、信息层级，但不要新增大量顶级页面。

## 4. 核心用户流程

请围绕以下流程重构体验：

1. 用户先在设置中配置 AI 模型与数据源。
2. 用户回到首页，输入股票代码或股票名称。
3. 系统创建分析任务，用户看到任务状态。
4. 分析完成后，用户阅读结构化报告。
5. 用户可以对报告进行重新分析、查看完整报告，或进入问股 Agent 继续追问。
6. 用户可以从历史记录中重新打开旧报告。
7. 用户可以回到设置页调整模型、数据源、通知和 Agent 配置。

设计重点是降低这个流程中的认知负担，让用户明确知道：

- 当前应该做什么。
- 系统正在做什么。
- 分析结果在哪里。
- 如何继续追问。
- 配置是否完整。

## 5. 页面功能说明

### 5.1 首页：股票分析工作台

首页是最重要的页面。

核心功能：

- 股票输入：支持代码和名称，例如 `600519`、`贵州茅台`、`AAPL`。
- 自动补全：输入时给出股票建议。
- 分析触发：用户确认股票后创建分析任务。
- 推送通知开关：用户可以决定本次分析是否触发通知。
- 分析任务展示：展示正在执行或排队中的任务。
- 历史记录：展示历史分析报告，支持选择、加载更多、批量选择和删除。
- 报告阅读：展示结构化分析报告。
- 报告操作：重新分析、追问 AI、查看完整 Markdown 报告。
- 状态反馈：空状态、加载中、任务重复、输入错误、API 错误。

设计方向：

- 首页应像“研究工作台”，不是欢迎页。
- 首屏应快速让用户开始分析或进入最近报告。
- 历史记录、任务状态、报告阅读之间的关系要更清晰。
- 报告区应适合长内容阅读，注意段落节奏、摘要信息、关键指标和风险提示。
- 移动端需要可用，但桌面端效率优先。

按钮设计要求：

- 请自行重新设计按钮布局位置。
- 只需区分主操作、次操作、危险操作、状态操作。
- 不要照搬当前按钮位置。
- 不要因为当前实现中某个按钮在某处，就认为它必须继续在那里。

### 5.2 问股 Agent 页面

问股页用于多轮策略问答。

核心功能：

- 会话列表。
- 新建会话。
- 删除会话。
- 技能/策略选择。
- 快捷问题。
- 用户输入问题。
- AI 流式回复。
- 推理/进度步骤展示。
- 消息复制。
- 会话导出 Markdown。
- 从报告页带股票上下文进入问股。
- 追问上下文加载状态。
- 错误状态和重试反馈。

设计方向：

- 问股页应像研究型 AI 助手，而不是客服聊天窗口。
- 需要兼顾“直接问问题”和“围绕某份报告继续追问”两种场景。
- 技能/策略选择应更容易理解，避免让普通用户觉得像技术参数。
- 快捷问题可以作为启动对话的辅助，而不是主视觉中心。
- 长回复、流式过程、思考步骤应有清楚层级。

按钮设计要求：

- 请重新设计发送、新建、导出、删除、复制等操作的位置和呈现。
- 删除类操作必须有确认。
- 复制、导出等工具操作不应抢占主输入体验。

### 5.3 设置页

设置页用于配置系统运行能力。当前已从平铺配置改成分组和抽屉式编辑。

设置分类：

- AI 模型。
- 数据源。
- 通知渠道。
- 系统设置。
- Agent 设置。
- 基础设置。

AI 模型当前方向：

- 只保留统一的 AI 模型接入。
- 标准模型服务统一使用 `base_url + api_key + model` 形式。
- Google AI Studio / Gemini 是特殊通道：可只配置 API Key 和模型，不需要 Base URL。
- 不再把 OpenAI、DeepSeek、Anthropic、AIHubmix、Gemini 等旧字段平铺展示。
- 不再暴露高级模型路由 YAML 作为普通用户入口。

数据源方向：

- 多数据源仍然重要，因为单一数据源可能不稳定或有错误数据。
- 配置应帮助用户理解“行情数据源”和“新闻搜索源”的区别。

通知方向：

- 保留必要通知渠道。
- Bot 机器人通知相关配置不应出现。

Agent 方向：

- 问股 Agent 是核心功能。
- Agent 设置需要更易理解，避免过度工程化。

设计方向：

- 设置页应降低配置焦虑。
- 用清晰分组、抽屉、状态提示、保存反馈来组织复杂配置。
- 模块内保存比全局保存更符合用户预期。
- 配置项需要说明“这是做什么的、什么时候需要配置、配置后如何验证”。
- 连接性测试能力应作为后续设计重点：AI 模型和数据源配置完成后，用户应能测试是否连通。

按钮设计要求：

- 请重新设计保存、测试连接、重置、打开配置、删除渠道等操作的布局。
- 本 brief 不指定这些按钮放在页面顶部、底部、抽屉内或卡片内。
- 请根据用户任务流重新规划。

### 5.4 登录页

核心功能：

- 当系统启用登录保护时，用户需要输入密码。
- 登录失败需要明确反馈。
- 加载状态和重试状态要清楚。

设计方向：

- 简洁、安全、可信。
- 不需要营销文案。
- 不需要复杂背景视觉。

## 7. 设计风格要求

整体风格：

- 专业。
- 克制。
- 稳定。
- 信息密度适中偏高。
- 适合长期反复使用。
- 更像研究工作台和分析工具，而不是营销站点。

避免：

- 大面积营销 hero。
- 过度装饰。
- 低信息密度卡片堆叠。
- 炫技动画。
- 单一色系铺满全站。
- 太多渐变和装饰光斑。
- 把功能说明写成大段教学文案堆在页面上。

建议：

- 保持明暗主题。
- 用颜色表达状态和优先级，而不是纯装饰。
- 报告阅读区要有良好排版。
- 配置页面要有强层级。
- 关键风险、错误、未配置状态要醒目但不刺眼。
- 长内容场景要考虑滚动、固定区域、状态恢复。

## 8. 状态设计要求

请覆盖这些状态：

- 空状态。
- 加载中。
- 分析中。
- 任务排队中。
- 流式回复中。
- 输入错误。
- 配置缺失。
- 连接测试成功。
- 连接测试失败。
- 保存成功。
- 保存失败。
- 删除确认。
- 批量选择。
- 禁用状态。
- 移动端导航展开。
- 桌面端侧边栏收起。

## 9. 响应式要求

优先级：

1. 桌面端研究效率。
2. 平板端可读性。
3. 手机端核心流程可用。

请为以下断点思考布局：

- 桌面宽屏。
- 普通笔记本。
- 平板。
- 手机。

不要只给桌面设计。首页、问股页和设置页都需要移动端可用方案。
