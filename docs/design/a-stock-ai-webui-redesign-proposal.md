# A_stock_AI WebUI Redesign Proposal

> 目标：为 `A_stock_AI` 前端重构提供一份可落地的 UI/UX 设计方案。本文适合交给 Figma / Claude Design 继续出图，也适合作为后续前端实现的设计依据。
>
> 设计方向：专业研究终端风 + 现代 AI 工作台交互。
>
> 重要约束：本文定义信息架构、页面结构、组件职责、状态和视觉规则，不强制规定具体按钮位置。按钮和操作区应在 Figma 设计阶段重新布局。

## 1. Design Principles

### 1.1 专业，而不是营销

这个产品是 A 股分析工作台，不是官网。首屏应直接服务“输入股票、查看任务、阅读报告、追问 AI”，不需要大 hero、宣传文案或装饰插图。

### 1.2 信息密度适中偏高

用户会反复查看报告、历史、任务和配置。界面应支持扫描、比较和长时间阅读，避免过大的卡片、过多留白和过强装饰。

### 1.3 清晰的任务链路

核心链路必须始终清楚：

1. 配置模型和数据源。
2. 输入股票。
3. 创建分析任务。
4. 阅读报告。
5. 追问 AI。
6. 回看历史。

### 1.4 AI 是研究助理，不是客服窗口

问股页要突出“围绕股票、报告、策略进行研究”，而不是普通聊天产品。技能、上下文、推理进度、引用来源都应该有明确层级。

### 1.5 设置页是控制台

设置页不应像字段清单。它应该帮助用户理解“我还缺什么配置、配置是否可用、保存后是否生效”。

## 2. Visual Direction

推荐风格：**Professional Research Console**。

### 2.1 视觉关键词

- 克制。
- 清晰。
- 稳定。
- 金融研究感。
- 强层级。
- 低装饰。
- 高可读性。

### 2.2 色彩建议

建议建立语义色板，而不是围绕单一品牌色铺满页面。

- Background：低饱和灰白 / 深灰黑，用于页面底色。
- Surface：卡片和面板，比背景略高一层。
- Border：轻边框，帮助区分信息区。
- Primary：用于主操作和当前焦点，建议偏蓝绿或青蓝，避免大面积紫蓝渐变。
- Positive：上涨、成功、连接正常。
- Negative：下跌、失败、危险操作。
- Warning：配置缺失、风险提示、任务异常。
- Info：AI 进度、提示、系统说明。

深色模式可以更像研究终端，但不要默认把全站做成“黑底霓虹”。浅色模式也必须完整可用。

### 2.3 字体与排版

- 页面标题：简洁，不要过大。
- 报告正文：优先可读性，行高充足。
- 数据标签：小字号但要清楚。
- 数字、代码、模型名、股票代码：可使用等宽字体。
- 避免负字距和视口宽度驱动字号。

### 2.4 形状与阴影

- 圆角控制在 8-12px；工具类控件可更紧凑。
- 阴影轻量，主要用于浮层、抽屉、弹窗。
- 主要分隔依靠边框、背景层级和间距，不靠大阴影堆叠。

## 3. Global Information Architecture

### 3.1 顶级导航

建议保留三大核心入口：

- **分析工作台**：股票分析、任务、历史、报告。
- **问股 Agent**：多轮问答、策略技能、报告上下文追问。
- **系统设置**：模型、数据源、通知、Agent 和基础配置。

登录页不作为主导航入口。

### 3.2 全局 Shell

建议继续采用侧边导航 + 顶部上下文栏的结构，但做信息收敛：

- 侧边导航只承担一级入口，不承载过多状态。
- 顶部栏显示当前页面标题、页面级状态、主题切换和必要系统入口。
- 移动端使用抽屉导航。
- 桌面端允许侧边栏收起。

### 3.3 全局状态入口

可以考虑在全局层提供轻量状态信号：

- 当前是否有任务运行。
- 模型/数据源配置是否缺失。
- 问股是否有新回复。

这些状态不应喧宾夺主，只作为工作台提醒。

## 4. Home Workspace

首页是核心页面，建议重构成“研究工作台”。

### 4.1 Primary User Tasks

- 快速输入或选择股票。
- 查看当前分析任务。
- 打开历史报告。
- 阅读结构化分析结果。
- 基于报告继续追问 AI。
- 对同一股票重新分析。

### 4.2 Proposed Layout

建议用三层工作台结构，而不是普通页面流：

**Workspace Header**

- 股票输入与自动补全。
- 当前分析状态。
- 与本次分析相关的轻量选项，例如是否通知。

**Research Sidebar**

- 当前任务。
- 历史记录。
- 批量选择与删除。
- 加载更多。

**Report Canvas**

- 报告摘要。
- 策略判断。
- 技术/资金/基本面/新闻等分区。
- 风险提示。
- 原始 Markdown 或完整报告入口。

注意：上述是区域职责，不是按钮位置规定。具体按钮和工具栏布局由 Figma 阶段重新设计。

### 4.3 Report Reading Model

报告展示建议分为四级：

1. **结论层**：评级、信心、核心摘要、关键风险。
2. **证据层**：技术面、资金面、基本面、新闻情报。
3. **策略层**：操作建议、条件触发、观察点。
4. **原文层**：完整 Markdown 报告。

这样用户可以先看结论，再决定是否展开细节。

### 4.4 States

首页至少需要这些状态：

- 未输入股票。
- 自动补全展开。
- 输入不合法。
- 分析任务创建中。
- 分析中。
- 任务重复。
- 历史为空。
- 报告加载中。
- 报告加载失败。
- 报告已生成。
- 删除历史确认。

### 4.5 Mobile Behavior

手机端建议将工作台压缩成分层导航：

- 顶部保留股票输入。
- 任务/历史作为可切换抽屉或标签。
- 报告作为主内容。
- 报告操作收进上下文菜单或底部操作区。

不要在手机端强行保留桌面三栏。

## 5. Ask-AI Agent Page

问股页应从“聊天页”升级成“股票研究助理”。

### 5.1 Primary User Tasks

- 直接问股票问题。
- 从报告进入并追问。
- 选择分析策略或技能。
- 查看 AI 推理/执行进度。
- 复制或导出结果。
- 管理历史会话。

### 5.2 Proposed Layout

建议拆成四个概念区：

**Session Rail**

- 会话列表。
- 新建会话。
- 删除会话。
- 当前会话状态。

**Research Context**

- 当前股票上下文。
- 来源报告。
- 当前技能/策略。
- 可切换或查看技能说明。

**Conversation Canvas**

- 用户消息。
- AI 回复。
- 推理进度。
- 错误提示。
- 流式加载状态。

**Composer**

- 问题输入。
- 快捷问题。
- 发送状态。

具体布局可以是左右分栏、上下分区或响应式抽屉，不在本文固定。

### 5.3 Skill Selection

当前技能名偏工程化。设计上建议转换成用户能理解的策略标签：

- 趋势分析。
- 缠论。
- 波浪理论。
- 箱体震荡。
- 情绪周期。
- 通用分析。

高级 ID 可隐藏在说明中，不应成为主要视觉元素。

### 5.4 Streaming And Thinking

流式进度应表达为研究过程，而不是技术日志：

- 正在识别股票。
- 正在读取历史报告。
- 正在调用行情数据。
- 正在分析技术结构。
- 正在生成结论。

用户应能折叠详细进度。

### 5.5 Mobile Behavior

手机端建议：

- 会话列表进入抽屉。
- 研究上下文用顶部 compact chip 展示。
- 输入区固定在可用位置。
- 长回复优先阅读体验。

## 6. Settings Page

设置页建议设计为“配置控制台”。

### 6.1 Category Model

建议保留分类：

- AI 模型。
- 数据源。
- 通知渠道。
- Agent 设置。
- 基础设置。
- 系统设置。

分类导航应说明每类是否已配置、是否有错误、是否有待保存项。

### 6.2 AI Model Configuration

AI 模型页重点是统一通道配置。

必须表达清楚：

- 标准服务：`base_url + api_key + model`。
- Google AI Studio / Gemini：特殊通道，可不填 Base URL。
- 支持多渠道 fallback。
- 支持 API Key 密文显示和切换可见。
- 支持连接测试。
- 支持模型列表发现或手动填写。

不要重新引入旧式平铺字段：

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `GEMINI_API_KEY`
- `DEEPSEEK_API_KEY`
- `ANTHROPIC_API_KEY`
- `AIHUBMIX_KEY`
- `LITELLM_CONFIG`

旧字段只应作为迁移来源或兼容层，不应成为 UI 概念。

### 6.3 Data Source Configuration

数据源配置建议分为：

- 行情数据源。
- 新闻搜索源。
- 优先级与兜底。
- 超时与重试。

设计重点：

- 告诉用户每个数据源解决什么问题。
- 展示当前是否配置、是否可测试。
- 多源优先级要可理解。

### 6.4 Notification Configuration

通知只保留必要渠道：

- 企业微信。
- 飞书。
- 邮件。
- PushPlus / ServerChan / Pushover / Webhook 等轻量推送。

不要出现 Bot 机器人通知。

### 6.5 Agent Configuration

Agent 设置建议弱化工程参数，强化用户任务：

- 是否启用问股 Agent。
- 默认分析策略。
- 多 Agent 深度。
- 超时预算。
- 风控策略。
- 记忆与校准。

每项要解释“什么时候需要改”。

### 6.6 Save / Test / Reset Interaction Model

建议将设置操作分成三类：

- **保存**：模块级保存，尽量靠近当前配置上下文。
- **测试连接**：只对模型、数据源、通知渠道等外部连接开放。
- **重置/清空**：需要明确风险提示。

不要依赖全局保存作为唯一入口。具体按钮位置由设计阶段决定。

### 6.7 Settings States

设置页至少覆盖：

- 未配置。
- 已配置。
- 待保存。
- 保存中。
- 保存成功。
- 保存失败。
- 测试中。
- 测试成功。
- 测试失败。
- 密钥已隐藏。
- 配置冲突。

## 7. Login Page

登录页保持简单。

### 7.1 Goals

- 安全可信。
- 快速进入系统。
- 错误反馈明确。

### 7.2 Suggested Structure

- 产品标识。
- 密码输入。
- 登录操作。
- 错误提示。
- 加载状态。

不需要介绍产品能力，不需要营销图，不需要复杂背景。

## 8. Button And Action System

本文不规定按钮具体摆放位置，只定义操作角色。

### 8.1 Action Roles

- **Primary**：开始分析、发送问题、保存配置、登录。
- **Secondary**：重新分析、追问 AI、查看完整报告、导出。
- **Tertiary / Ghost**：复制、展开、收起、刷新、打开详情。
- **Danger**：删除历史、删除会话、清空配置。
- **Stateful**：连接测试、任务状态、保存中、分析中。

### 8.2 Rules

- 每个区域只能有一个主操作。
- 危险操作必须二次确认。
- 工具操作不要抢主操作视觉权重。
- 图标按钮必须有 tooltip 或明确 aria label。
- 禁用按钮需要说明原因，尤其是配置缺失和任务运行中。

## 10. Implementation Phases

### Phase 1: Visual Foundation

- 统一色彩 token。
- 统一按钮、输入框、卡片、浮层。
- 收敛圆角、阴影、间距。
- 建立报告阅读排版规则。

### Phase 2: Home Workspace

- 重构首页布局。
- 优化股票输入、任务、历史和报告之间的关系。
- 优化报告摘要和报告操作层级。

### Phase 3: Ask-AI Agent

- 重构会话、上下文、策略选择和消息阅读体验。
- 优化流式进度和长回复阅读。

### Phase 4: Settings Console

- 优化分类状态。
- 优化抽屉配置。
- 增加连接测试交互设计。
- 优化模块级保存反馈。

### Phase 5: Mobile Polish

- 为首页、问股和设置提供手机端专用交互。
- 优化抽屉、底部操作、长内容滚动和输入体验。

## 11. Open Design Decisions

这些问题建议在 Figma 阶段探索 2-3 个方案：

- 首页是否采用桌面三栏工作台，还是“历史/任务侧栏 + 报告主画布”二栏结构。
- 问股页中策略选择是放在上下文面板，还是作为 composer 附近的轻量 chip。
- 设置页分类状态是放在左侧导航，还是在分类首页用概览卡表达。
- 深色模式是否作为默认演示风格，还是浅色优先。

## 12. Recommended Direction

推荐优先选择：

- 桌面端：研究工作台结构，信息密度偏高。
- 移动端：主任务优先，历史和设置通过抽屉承载。
- 视觉：浅色和深色都完整设计，深色可作为专业终端感展示，但不要只做深色。
- 首页：报告阅读体验优先。
- 问股：研究上下文优先。
- 设置：配置状态和连接测试优先。
